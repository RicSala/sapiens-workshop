# Sapiens Workshop — Roadmap & Progress Tracker

A Next.js app that generates deepdive courses on any topic for a specific audience. Courses can be read, listened to, exported as EPUB, and emailed.

---

## 1. Product shape

User flow:

1. **Form** — user enters topic, audience, tone, language, module count, words-per-module.
2. **Syllabus generation** — Claude returns `{ title, modules: [{ title, summary }] }`. User can edit/regenerate before committing.
3. **Module generation** — on approval, all modules generate in parallel; each appears as it finishes.
4. **Consumption** — course is readable as text, listenable as per-module audio (on-demand TTS), exportable as EPUB, and emailable.

No auth. Single user. The recipient email is configured in env.

## 3. Database schema (Approach A — normalized)

Decided: normalized schema. Per-row writes are essential for Slice 3 (parallel module generation). Audio bytes live in Vercel Blob, not in Postgres.

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}

enum CourseStatus {
  draft           // syllabus not yet generated
  syllabus_ready  // syllabus generated and (maybe) edited; modules pending
  generating      // module generation in progress
  ready           // all modules generated
  failed          // generation failed at some stage
}

enum ModuleStatus {
  pending     // not yet generated
  generating  // Claude call in flight
  ready       // content present
  failed      // generation failed
}

model Course {
  id                    String       @id @default(cuid())
  topic                 String
  audience              String
  tone                  String?      // optional, e.g. "casual", "academic"
  language              String       @default("en")
  targetModuleCount     Int
  targetWordsPerModule  Int
  title                 String?      // generated by Claude with the syllabus
  status                CourseStatus @default(draft)
  modules               Module[]
  createdAt             DateTime     @default(now())
  updatedAt             DateTime     @updatedAt

  @@index([createdAt])
}

model Module {
  id            String       @id @default(cuid())
  courseId      String
  course        Course       @relation(fields: [courseId], references: [id], onDelete: Cascade)
  order         Int          // 0-based position within the course
  title         String
  summary       String       // from the syllabus phase
  content       String?      // markdown, populated in Slice 3
  status        ModuleStatus @default(pending)
  errorMessage  String?      // populated when status = failed
  audio         ModuleAudio?
  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @updatedAt

  @@unique([courseId, order])
  @@index([courseId])
  @@index([status])
}

model ModuleAudio {
  id          String   @id @default(cuid())
  moduleId    String   @unique
  module      Module   @relation(fields: [moduleId], references: [id], onDelete: Cascade)
  blobUrl     String   // public URL returned by `put()` from @vercel/blob
  blobPathname String  // pathname under the blob store, kept so we can `del()` it
  mime        String   @default("audio/mpeg")
  durationMs  Int?     // optional, if we can extract it
  createdAt   DateTime @default(now())
}
```

Notes:

- `Course.title` is populated by Claude during syllabus generation, not by the user.
- `Module.order` is a plain int, not a fractional rank — reordering rewrites a few rows; fine at this scale.
- `Module.content` is markdown. Rendering happens client-side via a markdown component.
- `ModuleAudio.blobPathname` is stored separately from `blobUrl` because `@vercel/blob`'s `del()` takes pathnames; this saves us re-parsing the URL.
- Cascading deletes mean dropping a `Course` cleans up `Module` rows; dropping a `Module` cleans up its `ModuleAudio` DB row (but **does not** delete the blob — handled in app code, see Slice 6).

### Slice 1 — Syllabus generation (no persistence) `[x]`

**Goal:** prove the LLM call works end-to-end and we can render a coherent syllabus from a form, with content streaming into the UI as it generates.

**Scope**

- `/` is a landing page with a single CTA → `/courses/new`.
- `/courses/new` holds **only** the generation form (shadcn):
  - `topic` (text, required)
  - `audience` (textarea, required)
  - `tone` (text, default "neutral")
  - `language` (text, default "English")
  - `targetModuleCount` (number, default 10, range 3–20)
  - `targetWordsPerModule` (number, default 1000, range 200–3000)
- Dev-only pre-fill: when `NODE_ENV === "development"` the form ships with sample values so we can hit "Generate" without typing.
- On submit, the form validates client-side with zod, stashes the parsed input in `sessionStorage` (`lib/syllabus-draft-store.ts`), and `router.push('/courses/new/draft')`.
- `/courses/new/draft` reads the stashed input, streams the syllabus, and shows it:
  - Vercel AI SDK v6: POST `app/api/syllabus/route.ts` → `streamObject({ model, schema, system, prompt }).toTextStreamResponse()`
  - Client uses `experimental_useObject` from `@ai-sdk/react`; auto-fires `submit(input)` on mount once
  - Partial title/modules render as they stream in, with a pulsing indicator. Buttons: **Stop** (while streaming), **Edit prompt** (back to form), **Regenerate** (re-fire on the same input).
  - If the page is visited with no stashed input (direct nav / cleared session), shows a friendly empty state with a "Back to form" button.

**Verify**

1. Run `npm run dev`.
2. Open `/`, click "Start a new deepdive" → land on `/courses/new`.
3. Submit the form (dev defaults are pre-filled) → navigate to `/courses/new/draft`.
4. Watch the title appear first, then modules stream in one by one (title before summary). Pulsing dot while streaming; **Stop** cancels mid-stream.
5. Click **Edit prompt** to return to the form; click **Regenerate** to re-stream with the same input.
6. Visit `/courses/new/draft` directly in a fresh tab → empty-state card with a back link (expected).
7. Reload → still no persistence in the DB (Slice 2).

- `app/courses/new/draft/_components/syllabus-outline.tsx` — outline rendering (dnd-kit drag-to-reorder, hover-reveal `+` insertion zones, auto-grow textareas)

**Outline design (picked from 6 variants)**

- Single-column doc-like outline. Each row: drag handle (left gutter, opacity ramps on hover), inline-editable title + summary, hover-revealed delete (right).

- All edits run through the streamer's local-mirror state. **Regenerate** clears the mirror and re-fires the stream. Persistence comes in Slice 2.

### Slice 3 — Sequential module generation (streaming) `[x]`

**Decision:** generation is **sequential** with cumulative context, not parallel. Parallel calls would produce modules unaware of each other's content (redundant intros, no callbacks). Sequential trades wall-clock for coherence.

**Architecture:** idiomatic AI SDK pattern — one POST route handler streams a single module via `streamText(...).toTextStreamResponse()`. The **client** drives sequential generation by reading the stream of module N to completion before kicking off module N+1. No fire-and-forget, no polling, no server-side background work.

**Goal:** click "Generate modules" on the syllabus, watch modules stream live in the reader, one after another.

**Scope**

- Model: `claude-sonnet-4-6` (separate from the syllabus model). Set as `moduleModel` in `lib/ai.ts`.
- "Generate modules →" on the syllabus editor just navigates to `/courses/[id]`. The reader auto-starts streaming any non-ready modules on mount.
- **Route handler** `POST /api/courses/[id]/modules/[moduleId]/generate`:
  1. Loads course + prior `ready` modules.
  2. Marks `Module.status = generating` + `Course.status = generating`.
  3. Builds prompt = course context + full syllabus + prior modules' full content + current module instructions.
  4. Applies Anthropic `cacheControl: { type: "ephemeral" }` to the stable prefix so calls 2..N hit cache for the (course context + syllabus) portion.
  5. Calls `streamText` and returns `result.toTextStreamResponse()` (plain `text/plain` stream).
  6. `onFinish` persists final `content` + `status = ready`, then calls `maybeFinalizeCourse` to flip `Course.status` to `ready`/`failed` once nothing's pending.
  7. `onError` persists `status = failed` with the error message.
- **Streaming client** (`app/courses/[id]/_components/course-reader.tsx`):
  - On mount, iterates `initialModules` in syllabus order. For each non-ready module, POSTs the generate endpoint, reads the response body via `pipeThrough(new TextDecoderStream())`, and appends each chunk to that module's local content state.
  - Renders partial markdown live with a blinking cursor while the module is mid-stream.
  - On stream done: marks the module `ready` locally and moves to the next.
  - On error / network fail: marks the module `failed` locally with the error, **continues** with subsequent modules (so one bad module doesn't kill the whole run).
  - When the queue finishes, calls `router.refresh()` to confirm the canonical DB state.
- **Retry single module**: a Retry button on a failed module fires a single `streamOne(id)` call (same POST endpoint), bypassing the queue.

**Verify**

1. Open a saved course (status = `syllabus_ready`) and click "Generate modules →".
2. Page switches to the content view, status pill turns amber "Generating".
3. Each module transitions `pending → Writing → Ready`. Content appears as markdown one module at a time, in order. Total time ~30s × N.
4. Read module N — it references / builds on earlier modules without re-explaining their concepts.
5. Force a failure mid-generation (kill network, invalid env) → that module ends `failed` with an error, others keep running. "Retry" on the failed one re-runs it.

**Files**

- `lib/ai.ts` — added `moduleModel` (`claude-sonnet-4-6`)
- `lib/prompts/module.ts` — `MODULE_SYSTEM`, `buildModuleUserPromptParts` (returns `[stablePrefix, variableSuffix]` for cache-control placement)
- `lib/markdown.tsx` — `react-markdown` + `remark-gfm` wrapper; uses Tailwind `@plugin "@tailwindcss/typography"`
- `app/api/courses/[id]/modules/[moduleId]/generate/route.ts` — POST returns a `streamText(...).toTextStreamResponse()`. `onFinish`/`onError` persist final state. Also runs `maybeFinalizeCourse` to roll up the course-level status.
- `app/courses/[id]/layout.tsx` — shared layout: back link + `<CourseTabs />`
- `app/courses/[id]/_components/course-tabs.tsx` — pathname-based tab switch between "Course" and "Syllabus"
- `app/courses/[id]/page.tsx` — **course reader** (or empty-state CTA pointing to /syllabus when not yet generated)
- `app/courses/[id]/_components/course-reader.tsx` — streaming client; on mount, sequentially POSTs the generate endpoint per non-ready module and reads the response body
- `app/courses/[id]/syllabus/page.tsx` — **syllabus editor** route
- `app/courses/[id]/syllabus/_components/syllabus-editor.tsx` — editable outline; "Generate modules →" just navigates to the reader

**Routing**

- `/courses/[id]` — the course (reader / content view)
- `/courses/[id]/syllabus` — the syllabus (editor view)
- Top-of-page tabs let users jump between the two from anywhere.
- After "Save course" on the draft page, we redirect to `/courses/[id]/syllabus` (the editing surface).
- Clicking "Generate modules →" navigates to `/courses/[id]` so the user watches the reader fill in.

**Dependencies added**

- `react-markdown`, `remark-gfm`, `@tailwindcss/typography`

**Known limitation (flagged for later)**

- Generation is driven by the open browser tab. If the user closes the tab mid-stream, the in-flight module ends up `status = generating` in the DB (because `onFinish` never fires). On next page load the reader sees it's not `ready` and re-runs it. Good enough for local dev; for prod we'd want a job queue (Inngest / Trigger.dev) that owns the streaming server-side and writes back to the DB regardless of client presence.

**Out of scope**

- Cancellation mid-generation (closing the tab is the cancel for now).
- Whole-course retry (Retry is per-module).

### Slice 6 — Per-module audio (TTS, on-demand, cached on Vercel Blob) `[ ]`

**Goal:** listen to a module in the browser; cache so subsequent plays are instant.

**Scope**

- Each module card gets a "Listen" button when `status === 'ready'`.
- Server action `synthesizeModuleAudio(moduleId)`:
  - If a `ModuleAudio` row exists, return its `blobUrl`.
  - Else: call OpenAI TTS with the module's content (chunked if > model limit), get an mp3 `ArrayBuffer`, upload to Vercel Blob via `put(\`courses/<courseId>/modules/<moduleId>.mp3\`, buffer, { access: 'public' })`, then insert the `ModuleAudio` row with the returned `url` and `pathname`.
  - Return `{ url, mime }`.
- Client component swaps the "Listen" button for an `<audio controls src={url} />` once we have a URL.
- Deletion path: when a module or course is deleted, an action calls `del(pathname)` from `@vercel/blob` for any cached audio (Prisma's cascade only removes the row, not the blob).
- TTS chunking: OpenAI TTS has an input limit; for a 1000–2000 word module we may need to split, synthesize each part, and concatenate the mp3 bytes (mp3 frames are concatenable; we just join the buffers).

**Verify**

1. Click Listen on a module → small spinner, then audio starts playing.
2. Reload the page, click Listen again → plays immediately (from Vercel Blob, no new TTS call).
3. Delete the module → blob is gone (check Vercel dashboard / `list()`).

**Files**

- `prisma/migrations/<ts>_module_audio/` — adds `ModuleAudio`
- `lib/tts.ts` — `synthesize(text): Promise<Buffer>` (handles chunking)
- `lib/blob.ts` — thin wrappers around `put`/`del`/`head`
- `app/actions/synthesize-audio.ts`
- Update delete-course / delete-module actions to also `del()` the blobs

**Dependencies to install**

- `openai`
- `@vercel/blob`

**Needs from user**

- `OPENAI_API_KEY`
- `BLOB_READ_WRITE_TOKEN` (auto-injected on Vercel; for local dev, generate one in the Vercel dashboard under Storage → Blob)

**Out of scope**

- Whole-course concatenated audio.
- Voice/speed selection per course (we hardcode a sensible default; can become a course-level field later).
- Background pre-generation.

## 6. Conventions

- **No auth.** Recipient email is from env.
- **Style/tone/language** are per-course form inputs, not global config.
- **Generation defaults:** 10 modules × 1000 words. Form lets user override.
- **Next.js 16 specifics** (per `AGENTS.md`): use Server Actions / Server Functions and Route Handlers. Read `node_modules/next/dist/docs/` before introducing new patterns.
- **Don't write throwaway helpers.** If a slice needs a util, put it in `lib/` with a clear name.
- **Server Actions live in `app/actions/`.** Pages don't define inline actions unless trivially scoped.
- **Client islands live in `_components/`** colocated with the route that uses them.
- **Run `prisma migrate dev`** for every schema change, commit the migration file.
